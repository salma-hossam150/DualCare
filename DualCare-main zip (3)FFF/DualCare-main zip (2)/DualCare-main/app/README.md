# Building the DualCare Android APK (Flet)
link for the whole project after getting the apk and the apk "https://drive.google.com/drive/folders/15N83rmGUjvpoFs0pGlT1FWF9NtOsMB7i?usp=sharing"
This guide documents the exact steps used to build a working Android APK from the
DualCare Flet desktop app on Windows.

---

## 0. Before you start

Make sure you're working in the **correct project folder** — the one that contains:

```
app/
├── src/
│   ├── main.py
│   ├── engine.py
│   ├── database.py
│   └── models.py
├── pyproject.toml
├── uv.lock
└── run_app.py
```

`pyproject.toml` should contain:

```toml
[tool.flet.app]
path = "src"
```

⚠️ **Common mistake:** some extracted copies of this repo also contain an empty
`my-app/` subfolder generated by `uv init` (a "Hello, world!" placeholder with no
Flet code). Building from `my-app/` produces a blank APK with no UI and no Firebase
connection. Always confirm `src/main.py` is large (15KB+) and imports `flet` before
building — that's your real app.

---

## 1. Open PowerShell in the project root

```powershell
cd path\to\DualCare\app
```

Confirm you're in the right place:

```powershell
dir
```

You should see `src`, `pyproject.toml`, `uv.lock`, etc.

---

## 2. Make sure Flet (with build tools) is installed

```powershell
uv add "flet[all]"
uv sync
```

`uv sync` installs everything declared in `pyproject.toml` into a local `.venv`.

You do **not** need to manually activate the venv — `uv run` always uses the
project's own `.venv` automatically. If you see this warning, it's harmless and can
be ignored:

```
warning: `VIRTUAL_ENV=.venv` does not match the project environment path ... and will be ignored
```

---

## 3. Run the build

```powershell
uv run flet build apk -vv
```

`-vv` gives verbose output, which is very helpful for spotting exactly where things
go wrong.

### What happens on a first run (in order)

1. **Flutter SDK download** (~1GB) — auto-installed to `%USERPROFILE%\flutter\<version>`
2. **JDK 17 download** — auto-installed to `%USERPROFILE%\java\<version>`
3. **Android SDK setup** — auto-installed/verified under
   `%USERPROFILE%\AppData\Local\Android\Sdk`
4. **Android SDK license prompts** — you must accept each one (see §4)
5. **"Creating app shell..."** — clones the Flet build template from GitHub (needs `git`)
6. **"Packaging Python app..."** — bundles your `src/` code + dependencies for Android
7. **"Building .apk for Android..."** — Gradle compiles everything; on a first run this
   also downloads the entire Kotlin/Gradle toolchain (dozens of small jars), which is
   by far the slowest one-time step
8. **Build complete** — APK copied to `build\apk\`

**Total time on a first build:** 30–50 minutes is normal, almost entirely from the
one-time Flutter/Gradle/Kotlin downloads. Every build after this one is much faster
since everything is cached on the machine.

---

## 4. If prompted to accept Android SDK licenses

If `flutter doctor` reports `Some Android licenses not accepted`, run:

```powershell
C:\Users\<you>\flutter\<version>\bin\flutter.bat doctor --android-licenses
```

Type `y` and press Enter for **each** license shown (usually 5–7 of them).

---

## 5. If `git` is missing

The build template clone step requires git. If you see `'git' is not installed`:

```powershell
winget install --id Git.Git -e --source winget
```

Then **close and reopen PowerShell** (PATH won't update in the current window),
`cd` back into the project, and re-run the build.

---

## 6. Find the finished APK

```powershell
dir build\apk
```

You should see something like:

```
dualcare.apk        (~65–70 MB)
dualcare.apk.sha1    (checksum file — not needed for install)
```

---

## 7. Install it on an Android phone

**Option A — USB cable + ADB** (fastest if set up):

```powershell
adb devices
adb install build\apk\dualcare.apk
```

**Option B — copy the file directly**

1. Connect phone via USB (set to "File Transfer" mode)
2. Copy `dualcare.apk` to the phone's Downloads folder
3. On the phone: open Files app → Downloads → tap `dualcare.apk`
4. Allow "install from unknown sources" if prompted, then confirm install

**Option C — share without a cable**

Upload to Google Drive / send via WhatsApp / email it to yourself, then open the file
on the phone to install.

The APK is self-contained and signed with the debug key — anyone can install and run
it on their own phone for testing/demo purposes. It only needs proper release signing
if you intend to publish to the Google Play Store.

---

## 8. After installing — what to check

- Does the UI render correctly (movement history chart, vitals, care articles)?
- Does it actually pull live data from Firebase, or stay empty?
- Does the background Firebase-polling logic (the `threading` use in `engine.py`)
  behave the same on the phone as it does on desktop?
